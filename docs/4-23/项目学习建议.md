> 生成日期：2026-04-23  
> 适用对象：想系统学习这个项目的开发者，尤其是想理解 **Next.js 16 + React 19 + AI SDK v6 coding agent 工作流** 的同学。

这份文档不是 README 的重复版，而是一份学习路线图。你可以把它当作“带任务的源码阅读指南”：先建立项目心智模型，再按链路阅读源码，最后通过几个小练习把知识串起来。

## 1. 先用一句话理解这个项目

`ai-sdk-demo` 不是普通聊天应用，而是一个 **本地代码仓库工程师 Agent 控制台原型**。

它把下面几件事放在同一个 Next.js 项目里演示：

- 用户选择本地工作区。
- 前端用 `useChat` 和服务端 Agent 流式对话。
- 服务端 Agent 可以读取目录、搜索代码、读取文件。
- Agent 可以提出写入文件请求，但默认必须经过前端审批。
- 对复杂调查任务，可以让只读 subagent 单独读很多文件，再把摘要交回主 Agent。
- 对大任务，可以先用 Plan 模式生成结构化计划，用户确认后再执行。
- 对长对话，可以把历史消息持久化到 SQLite，并在上下文过长时做摘要压缩。
- 可以通过 MCP 动态接入外部工具，这里用自写 weather server 做示例。

学习这个项目时，建议不要只把它看成“聊天 UI”，而要把它看成一个微型 coding agent runtime。

## 2. 学习前准备

### 2.1 需要先知道的技术

如果你还不熟悉这些概念，建议边读边补，不需要一次性全部学完：

- **Next.js App Router**：理解 `app/page.tsx`、`app/api/**/route.ts`、Server/Client Component 的边界。
- **React Hooks**：重点看 `useState`、`useEffect`、`useMemo`、`useRef` 如何一起管理会话和流式 UI。
- **AI SDK v6**：重点理解 `useChat`、`DefaultChatTransport`、`ToolLoopAgent`、`tool`、`streamObject`、approval、UIMessage parts。
- **Zod**：这个项目用它定义 route 输入、工具输入、Plan 输出 schema。
- **Node.js 文件系统能力**：`node:fs`、`node:path`、子进程、路径安全校验。
- **SQLite / better-sqlite3**：用于保存聊天消息和压缩摘要。
- **MCP 基础概念**：server 暴露工具，client 在请求时动态拉起并合并工具集。

### 2.2 特别提醒：Next.js 版本

项目根目录的 `AGENTS.md` 已经明确提醒：

这个项目使用 `next@16.2.3`，并且项目要求在写 Next.js 相关代码前先阅读 `node_modules/next/dist/docs/` 中相关指南。学习时可以先按源码理解；真正修改 Next.js 路由、渲染、配置、缓存或构建行为前，再查对应本地文档。

## 4. 项目心智模型

学习时建议记住这个主链路：

```text
浏览器
  ↓
app/page.tsx
  ↓
useChat + DefaultChatTransport
  ↓
POST /api/chat
  ↓
createProjectEngineerAgent()
  ↓
createChatAgent()
  ↓
prepareCall()
  ├─ 校验 workspaceRoot
  ├─ 拼 persona / developer rules
  ├─ 收集 environment_context
  ├─ 收集 AGENTS.md 指令
  └─ 注入 experimental_context
  ↓
ToolLoopAgent
  ├─ list_files / search_code / read_file
  ├─ write_file / edit_file
  ├─ explore_workspace
  ├─ ask_question / ask_choice / show_reference
  ├─ update_plan
  └─ weather MCP tools
  ↓
流式 UIMessage 返回前端
  ↓
MessageBubble / ToolPartCard 渲染文本、工具状态和审批卡
```

这个模型里最重要的是四个边界：

- **UI 边界**：前端只负责会话、输入、展示、审批交互，不直接读取本地文件。
- **Route 边界**：`/api/chat` 接收前端请求，负责创建本轮 Agent、合并工具、保存消息。
- **Agent 边界**：`ToolLoopAgent` 负责判断何时调用工具、何时继续生成回答。
- **Workspace 边界**：所有文件读写都必须通过 `workspaceRoot` 校验，防止路径逃逸。

### 阶段 2：读懂前端主页面

目标：理解 `app/page.tsx` 为什么是“状态编排中心”。

### 阶段 3：读懂主聊天路由

目标：理解一次用户消息从 HTTP 请求进入服务端后如何变成 Agent 执行。

核心文件是 `app/api/chat/route.ts`。建议按代码顺序读：

1. 检查 gateway API key。
2. 解析请求 body：消息、会话 ID、工作区、访问模式、审批模式。
3. 根据 access mode 判断是否开放 workspace tools。
4. 尝试拉起 weather MCP tools，失败时降级继续。
5. 清洗前端传来的 UIMessage。
6. 加载已有 summary，并判断是否需要 compaction。
7. 创建主 Agent。
8. 注册 active stream，用于刷新或切换后的流式恢复。

### 阶段 5：读懂只读工具系统

目标：理解 Agent 如何安全读取工作区。

核心文件：

- `lib/workspace-tools.ts`
- `lib/workspaces.ts`
- `lib/tool-result.ts`

只读工具有三个：

- `list_files`：受限目录遍历。
- `search_code`：用 `@vscode/ripgrep` 搜索代码，失败时回落到 Node 搜索。
- `read_file`：读取 UTF-8 文本文件，二进制文件会省略，长文件会截断。

最关键的是路径安全：

- 所有相对路径都会通过 `resolveWorkspacePath()` 解析。
- 如果路径试图通过 `..` 逃出 `workspaceRoot`，会直接报错。
- 遍历和搜索会跳过 `.git`、`.next`、`node_modules`、`dist`、`build` 等高噪声目录。

本阶段重点问题：

- `getWorkspaceToolContext()` 如何从 AI SDK 的 `experimental_context` 中取出运行时上下文？
- `listWorkspaceEntries()` 如何限制 depth 和 limit？
- `searchWorkspace()` 为什么优先用 `@vscode/ripgrep` 而不是系统 `rg`？
- `readWorkspaceFile()` 如何识别二进制和截断长文本？
- `ToolResult<T>` 为什么要做成 discriminated union？

建议练习：

- 找到 `search_code` 的 Zod input schema，解释每个字段如何限制工具调用范围。
- 手动构造一个试图访问 `../` 的路径，推演它会在哪里被拒绝。
- 修改 `IGNORED_DIRECTORY_NAMES` 的含义说明，思考新增忽略目录会影响哪些工具。

### 阶段 6：读懂写入工具和审批机制

目标：理解项目如何让 Agent “能改文件但不乱改文件”。

核心文件：

- `lib/write-tools.ts`
- `lib/tool-helpers.ts`
- `app/_components/tool-card/interactive-cards.tsx`
- `app/_components/tool-card/ToolPartCard.tsx`

写入工具有两个：

- `write_file`：创建文件或完整覆盖文件。
- `edit_file`：对已有文件做精确 search-replace。

设计重点：

- 默认所有写入都需要用户审批。
- `bypassPermissions` 为 `true` 时才跳过审批。
- 写入工具仍然复用 `resolveWorkspacePath()`，不能逃出工作区。
- `edit_file` 要求 `oldString` 精确匹配，默认必须唯一。
- `write_file` 的描述明确禁止主动创建文档，除非用户明确要求。本次你要求生成文档，所以这个场景是允许的。

本阶段重点问题：

- `needsApproval` 是在哪里决定的？
- `execute` 为什么只在审批之后才真正落盘？
- `edit_file` 为什么要求先读文件再改？
- `replaceAll` 的风险在哪里？
- 如果 `oldString` 匹配多次但没有设置 `replaceAll`，工具会怎么处理？

建议练习：

- 读懂 `editFileTool` 的完整错误分支。
- 画出“模型请求写入 -> 前端显示审批 -> 用户批准 -> 工具执行 -> 模型继续回复”的时序图。
- 思考如果要支持 diff preview，应该在 `write-tools` 层做，还是在前端 tool-card 层做。

### 阶段 8：读懂 subagent

目标：理解为什么要把“大范围代码调查”从主上下文隔离出去。

核心文件：

- `lib/subagents/explorer.ts`

`explore_workspace` 是一个工具，但工具内部会启动另一个 `ToolLoopAgent`。这个子 Agent 只能使用只读工具：

- `list_files`
- `search_code`
- `read_file`

它的价值是：

- 主 Agent 不必亲自读取 20 到 30 个文件。
- 子 Agent 可以在独立上下文里调查。
- 最终只把不超过 500 字的中文摘要和读过的文件列表返回给主 Agent。
- 主上下文不会被大量源码内容污染。

本阶段重点问题：

- `explorerAgent.generate()` 和主聊天流式响应有什么区别？
- `filesExamined` 是如何从 tool calls 中提取出来的？
- 为什么 explorer 不能反问用户？
- 哪些问题应该用 explorer，哪些不应该？

建议练习：

- 找一个问题，例如“这个项目如何保存聊天历史”，判断主 Agent 应该直接读文件还是调用 explorer。
- 调整 explorer 的 step limit 会产生什么影响？
- 思考如果未来增加 `security_auditor` subagent，它应该有哪些工具和规则。

### 阶段 10：读懂持久化、恢复和压缩

目标：理解流式聊天应用在“刷新页面、切会话、长对话”场景下如何保持稳定。

核心文件：

- `lib/chat-store.ts`
- `lib/active-streams.ts`
- `app/api/chat/history/route.ts`
- `app/api/chat/[chatId]/stream/route.ts`
- `lib/compaction.ts`

这个项目里有三套相关机制：

1. **消息持久化**
   - SQLite 文件位于 `.data/chat.db`。
   - `messages` 表按 session 保存 UIMessage。
   - `saveMessages()` 每次保存会用 transaction 整段 replace。

2. **流式恢复**
   - POST `/api/chat` 开流时注册 active stream。
   - `consumeSseStream` 把 SSE chunk 复制到内存 buffer。
   - GET `/api/chat/[chatId]/stream` 可以 replay 已有 chunks 并继续订阅 live chunks。
   - 这个设计只适合本地单进程，多实例部署需要 Redis pub/sub 或其他外部协调。

3. **上下文压缩**
   - 每次请求前估算当前 agent message tokens。
   - 超过阈值后调用 `compactMessages()`。
   - 数据库保留完整 UI 历史。
   - Agent 只看摘要加最近消息。
   - 前端会看到一条 role 为 `system` 的压缩通知，但这条消息不会喂回 Agent。

本阶段重点问题：

- 为什么 session 元数据放 localStorage，而消息正文放 SQLite？
- 为什么 `activeStreams` 是进程内 Map，它的局限是什么？
- compaction 后为什么不能直接丢弃 DB 里的老消息？
- `compacted_count` 有什么调试价值？

建议练习：

- 画出“刷新页面但服务端流还在继续”的恢复流程。
- 找到 `estimateTokens()` 的实现，判断它是精确 token 还是粗略估算。
- 思考如果要部署到 Vercel，多实例下哪些部分需要重做。

## 8. 动手练习建议

### 练习 1：给只读工具增加一个轻量能力

目标：熟悉工具定义、Zod schema、ToolResult。

可选任务：

- 增加一个 `file_stats` 工具，输入 `relativePath`，返回文件大小、行数、是否二进制、是否被忽略。
- 或者给 `read_file` 增加 `startLine` / `endLine` 能力，避免每次读取整个文件。

需要关注：

- 路径必须经过 `resolveWorkspacePath()`。
- 返回值建议用 `toolOk()` / `toolErr()`。
- schema 描述要让模型知道什么时候该用这个工具。

### 练习 2：增强 Plan 输出

目标：熟悉 `streamObject` 和客户端 partial object。

可选任务：

- 给每个步骤增加 `verification` 字段。
- 给 Plan 增加 `assumptions` 字段。
- 在 PlanCard 中显示“验证方式”。

需要关注：

- `lib/plan-schema.ts` 要改。
- `lib/plan-generator.ts` 的 persona 要改。
- `PlanCard` 的 `EditableStep` 和 merge 逻辑要改。
- 接受计划转 markdown 的逻辑要改。

### 练习 3：优化工具输出 UI

目标：熟悉 UIMessage parts 和 tool-card 渲染。

可选任务：

- 对 `search_code` 输出按文件分组展示。
- 对 `read_file` 输出显示是否被截断。
- 对 `write_file` 审批卡增加更清晰的风险提示。

需要关注：

- 不要把工具结果的所有细节都塞进 UI。
- 工具输出需要“可读”和“足够透明”之间做平衡。
- approval UI 最重要的是让用户知道即将发生什么。

### 练习 4：增加一个新的 subagent

目标：理解 subagent 的上下文隔离价值。

可选任务：

- 增加 `security_review_workspace`，只读调查潜在安全风险。
- 增加 `test_planner`，专门根据代码变更建议测试策略。

需要关注：

- 子 Agent 的工具权限应该比主 Agent 更窄。
- 子 Agent 输出要短，不要把所有文件内容带回主上下文。
- 主 Agent 的工具 description 要清楚写明什么时候用、什么时候不用。

### 练习 5：把 active stream 改造成可部署设计

目标：理解本地原型和生产系统之间的差异。

可选任务：

- 设计一个 Redis pub/sub 版本的 `active-streams`。
- 或调研 `resumable-stream` 类库能否替代当前 Map。

需要关注：

- 当前 `activeStreams` 是进程内 Map，只适合本地单进程。
- 多实例下 POST 和 GET 可能落到不同进程。
- 要考虑 stream buffer 的过期、内存、错误传播。

## 9. 学习时容易踩的坑

### 坑 1：把它当普通 ChatGPT 壳子

这个项目重点不是消息列表，而是 Agent 工作流。读代码时要一直关注：

- prompt 如何构造；
- 工具如何定义；
- 工具如何审批；
- 上下文如何控制；
- 文件系统边界如何保证安全。

### 坑 2：只读前端，不读服务端

前端 UI 很重要，但核心逻辑在服务端：

- Agent 创建在 `/api/chat`。
- 工具执行在 `lib/*tools.ts`。
- prompt 构造在 `lib/chat-agent/*` 和 `lib/session-primer.ts`。
- 持久化在 `lib/chat-store.ts`。

### 坑 3：忽略 `experimental_context`

很多工具能正常工作的关键，是 `experimental_context` 里有：

- `workspaceRoot`
- `workspaceName`
- `workspaceAccessMode`
- `bypassPermissions`

如果你以后新增工具，首先要想清楚它需要哪些运行时上下文。

### 坑 4：忽略消息清洗和 compaction

流式 tool call 会产生很多中间态，如果刷新页面、切会话、重发请求时不清理，很容易出现 provider 报错。`sanitizeChatUIMessages()` 和 compaction 不是边角功能，而是让长对话稳定运行的重要工程细节。

### 坑 5：随手改 Next.js 相关代码

这个项目使用 Next.js 16。涉及 route handler、streaming、构建配置、缓存、服务端行为时，不要完全依赖旧经验。项目要求先阅读 `node_modules/next/dist/docs/` 中相关文档。

## 12. 下一步建议

建议你按下面顺序继续推进：

1. 先用 1 到 2 小时读完 `README.md`、`app/page.tsx`、`app/api/chat/route.ts`，建立主链路。
2. 再用 1 天读工具系统，重点看路径安全和 approval。
3. 然后选择一个小练习实现，例如给 Plan 增加 `verification` 字段，或者给 `read_file` 增加按行读取。
4. 修改前先查本地 Next.js 16 文档，尤其是当你要动 `app/api/**/route.ts` 或 streaming 行为时。
5. 每次改动后优先跑轻量验证，例如 `npm run lint`。只有改到 framework、路由、构建或服务端渲染关键行为时，再考虑 `npm run build`。

这个项目最值得学习的地方，是它把“模型能力”变成了一个可控的工程系统：有上下文边界、有工具权限、有审批、有持久化、有恢复、有长上下文治理。读源码时一直围绕这些工程问题思考，会比单纯记 API 更有收获。
